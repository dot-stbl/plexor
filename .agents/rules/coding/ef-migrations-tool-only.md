# EF Core migrations — created by `dotnet ef` ONLY

## Hard rule

> **Agent MUST NOT author, edit, or create EF Core migration files by hand.**
> The only sanctioned way to create or delete a migration is:
>
> ```bash
> dotnet ef migrations add <Name> \
>     --context <ContextName> \
>     --project <Module>.Infrastructure \
>     --startup-project <Migrator or Host>
> ```
>
> and the inverse:
>
> ```bash
> dotnet ef migrations remove \
>     --context <ContextName> \
>     --project <Module>.Infrastructure \
>     --startup-project <Migrator or Host>
> ```

The companion rule `.agents/rules/coding/ef-migrations-are-tool-generated.md`
explains *why* (the model snapshot coupling). This rule is the operational
prohibition — it lists what an agent must NOT do under any circumstance.

## What an agent MUST NOT do

- ❌ Hand-write a new migration file under `**/Migrations/**/*.cs`
- ❌ Edit the `Up()` / `Down()` body of an existing migration
  (including adding custom SQL for migrations that target a fresh schema)
- ❌ Touch the `*.Designer.cs` files generated by `dotnet ef`
- ❌ Hand-edit the `*ModelSnapshot.cs` files generated by `dotnet ef`
- ❌ Copy-paste a migration file from another project / branch
- ❌ "Fix" a broken migration by editing its body — instead, run
  `dotnet ef migrations remove` (which cleans up the migration AND
  its Designer) and `dotnet ef migrations add` again
- ❌ Add a `namespace` import, comment, or whitespace-only change to
  any migration file. EF Core's snapshot diff is whitespace- and
  comment-sensitive; cosmetic edits invalidate the snapshot

If the agent thinks it needs to do any of the above, the right path is
to use `dotnet ef migrations remove` (cleanly deletes both `.cs` and
`.Designer.cs`) followed by `dotnet ef migrations add NewName`. The
tool's body is what counts; the agent does not improve on it by hand.

## Self-test before commit

Run this against any staged migration file BEFORE the commit:

```bash
git diff --cached -- 'src/**/Migrations/**/*.cs' 'src/**/Migrations/**/*.Designer.cs' \
    | rg -q '^[-+].*' \
    || { echo 'No tool-generated changes — refusing to commit'; exit 1; }
```

If this prints nothing, the agent is hand-editing. The rule blocks
the commit before it lands in the repo.

## The one legitimate exception

There is exactly one reason to hand-edit a migration: EF Core cannot
express the schema the module needs (e.g. Postgres `CREATE INDEX
CONCURRENTLY`, custom operators, partial indexes). When this comes
up:

1. Comment at the top of the `Up()` body explaining which EF
   limitation this works around (in plain language, not just an
   EF Core error).
2. Run `dotnet ef migrations add` to seed the file. EF's migration
   name / ordering / snapshot stays correct.
3. The custom SQL block sits next to the explanation comment so a
   later reader can see "this is a workaround" without grepping
   commit history.
4. The body outside the SQL block still reflects what the model
   looks like — e.g. don't change a column type by hand without
   also changing the matching `HasColumnType("...")` in the
   DbContext's `IEntityTypeConfiguration`.

If you can't do all four, the migration must come from the tool —
no exceptions.

## Recovery when this rule was violated

See `.agents/rules/coding/ef-migrations-are-tool-generated.md` §"How
to recover when this rule was violated". The summary:

1. `git rm` the offending `Migrations/**/*.cs` and
   `Migrations/**/*.Designer.cs` files.
2. Delete the matching `*ModelSnapshot.cs`.
3. Make sure the DbContext is in the state the migrations were
   supposed to produce (i.e. apply any pre-existing hand-edits
   to the DbContext + entity configurations first).
4. Run `dotnet ef migrations add <Name> --context <Context> --project
   <Module>.Infrastructure --startup-project <Migrator>` to
   regenerate against the corrected model.
5. Diff the regenerated files against the deleted ones — they
   should encode the *intent* of the original hand-edit, just
   authored by `dotnet ef` not by an LLM.
